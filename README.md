一、GPU渲染管线
我理解就是一个“把3D模型变成屏幕图像”的流水线，分几步走：
1. 输入装配——把模型的顶点数据（位置、颜色、法线这些）准备好，告诉GPU“我要画这些点”。
2. 顶点着色器——这个是我可以写的程序。它的工作是：把每个顶点的3D坐标，通过矩阵变换，算出它在屏幕上该在哪个位置。简单说就是“把模型摆到镜头前”。
3. 图元装配——把顶点连成三角形。比如三个顶点连成一个三角形面，这一步是硬件自动做的。
4. 光栅化——把三角形“像素化”。算出来这个三角形覆盖了屏幕上哪些格子，生成一堆“片元”。
5. 片元着色器——这是我写的另一个程序。每个片元进来，我算它的颜色（光照、纹理贴图这些），输出一个颜色值。
6. 输出合并——做各种测试（深度测试看看有没有被挡住、模板测试之类的），最终把颜色写到屏幕上。
我的理解是：顶点着色器管“位置在哪”，片元着色器管“长什么样”。
二、常量缓冲区的更新方式
常量缓冲区，我理解就是“给着色器传全局参数的一个内存块”。比如MVP矩阵、光源位置、相机位置这些。
我学的更新方式有这几种：
1. Map/Unmap方式
用Map函数把GPU内存映射到CPU能访问的地址，填完数据，再Unmap。每次更新都要做这一步。
2. 每帧更新一次
适合相机矩阵这种一帧都不变的数据。我一般在帧开始的时候更新一次，所有物体共用。
3. 每物体更新一次
每个模型有自己的世界矩阵，画100个物体就要更新100次。这个比较麻烦，我学到的一个优化是环形缓冲区：提前申请一大块内存，每帧轮着用不同区域，避免CPU等着GPU读完才能写。
4. 局部更新
如果只改矩阵里的某一项，可以用UpdateSubresource这类函数只更新一部分，不用整个重传。
我踩过的坑：一开始我每个物体都单独Map一次，结果帧率掉得厉害。后来才知道常量缓冲区要尽量合并上传，减少CPU和GPU的同步次数。
三、图元、片元、像素的区别
这三个我刚开始完全搞混，后来是这么区分的
图元——就是三角形（或者线、点）。模型就是一大堆三角形拼起来的，每个三角形叫一个图元。
片元——光栅化阶段生成的“候选像素”。三角形覆盖到屏幕上，每个被覆盖的屏幕格子生成一个片元。每个片元带着深度值、插值后的纹理坐标这些信息。片元着色器就是对着片元算颜色的。
像素——最终显示在屏幕上的那个点。片元经过深度测试、混合这些环节，确定要显示，才变成像素。
我的理解打个比方：
图元 = 一张贴纸（三角形形状）
片元 = 把贴纸往墙上按，每个可能贴到的位置
像素 = 最终贴上去的那个点
片元不一定变成像素，比如深度测试不通过（被前面物体挡住了），这个片元就被扔掉了。
这三块内容，我觉得最绕的是管线流程，一开始总记不住顺序，后来就记“顶点→图元→光栅化→片元”这个链条
图元/片元/像素这个区别，面试问得挺多的，我的记忆口诀是：图元是几何，片元是候选，像素是结果
